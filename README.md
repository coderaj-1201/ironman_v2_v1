# RAG Enterprise — Microsoft Agent Framework

Multi-agent RAG system on Azure using **Microsoft Agent Framework (MAF) v1.0**.

## Architecture

```
Teams Bot (port 3978)
    │  HTTP
    ▼
Main Agent (port 8000)          ← MAF @workflow + @step
    │  HTTP                       /query  /feedback  /history  /memory
    ▼
Orchestrator Agent (port 8001)  ← MAF @workflow + @step
    │                                  │  classify query (domain + tool) + long-term memory
    │                                  │  retry loop ≤ 3 times
    │                                  │  tool escalation: hybrid → hyde → decomposition
    │  HTTP (local) / Service Bus (prod)
    ▼
Retrieval Agent (port 8002)     ← MAF @workflow + @step
    │  hybrid_search (BM25 + vector + RRF)
    │  hyde (hypothetical document generation)
    │  decomposition (sub-question fan-out)
    │  short-term memory injected into synthesis
    ▼
Azure AI Search (idx-rag)
    + Azure OpenAI (synthesis + confidence scoring)
    + Azure Cosmos DB (conversations, feedback, memory)
```

**Confidence loop:** If retrieval confidence < 0.75, orchestrator retries with next tool.
After 3 failed attempts, main agent returns failure + two escalation options:
- `raise_ticket` — creates ITSM ticket (ServiceNow stub)
- `connect_sme` — routes to Subject Matter Expert

## IDs

Every interaction carries a full set of traceable IDs:

| ID | Generated by | Purpose |
|---|---|---|
| `question_id` | `UserQuery` (auto `q-xxxxxxxx`) | Unique per question; Cosmos partition in conversations |
| `answer_id` | `FinalResponse` (auto `ans-xxxxxxxx`) | Links answer to feedback |
| `conversation_id` | Caller (or auto UUID) | Groups turns in a session |
| `user_id` | Caller | User identity |
| `feedback_id` | `FeedbackRequest` (auto `fb-xxxxxxxx`) | Unique per feedback submission |
| `memory_id` | `MemoryRecord` (auto `mem-xxxxxxxx`) | Unique per long-term memory fact |

## Cosmos DB Containers

| Container | Partition Key | Contents |
|---|---|---|
| `conversations` | `/conversation_id` | All Q&A turns with full metadata |
| `feedback` | `/user_id` | User ratings + comments per answer |
| `memory` | `/user_id` | Long-term user facts (extracted by LLM) |

Short-term memory (last N turns) is derived from `conversations` — no extra container.

## API Endpoints

### Main Agent (port 8000)

| Method | Path | Description |
|---|---|---|
| `GET` | `/health` | Health check |
| `POST` | `/query` | Ask a question |
| `POST` | `/feedback` | Submit feedback on an answer |
| `GET` | `/history/{conversation_id}` | Get conversation history |
| `GET` | `/memory/{user_id}` | Get long-term memory for a user |

#### POST /query
```json
// Request
{ "text": "What is the annual leave policy?", "conversation_id": "conv-123", "user_id": "alice" }

// Response
{ "reply": "Full-time employees...", "question_id": "q-abc12345", "conversation_id": "conv-123" }
```

#### POST /feedback
```json
// Request
{ "answer_id": "ans-def456", "question_id": "q-abc12345", "user_id": "alice", "rating": 5, "comment": "Very helpful" }

// Response
{ "status": "saved", "feedback_id": "fb-ghi789" }
```

#### GET /history/{conversation_id}?limit=10
```json
{ "conversation_id": "conv-123", "turns": [...], "count": 3 }
```

#### GET /memory/{user_id}
```json
{ "user_id": "alice", "memories": [{"memory_id": "mem-abc", "content": "User is in HR department"}], "count": 1 }
```

## Quick Start (Local — no Docker)

```bash
# 1. Install
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 2. Configure
cp .env.template .env
# Fill in AZURE_FOUNDRY_PROJECT_ENDPOINT, AZURE_SEARCH_ENDPOINT, AZURE_SEARCH_API_KEY
# For Cosmos features: fill in AZURE_COSMOS_ENDPOINT + AZURE_COSMOS_KEY
# (or run provision_cosmos.py — see below)

# 3. Provision Cosmos DB (first time only)
python scripts/provision_cosmos.py
# Prints the COSMOS_ENDPOINT and COSMOS_KEY to add to .env

# 4. Create search index + seed data
python scripts/create_search_index.py
python scripts/seed_data.py

# 5. Start all agents (3 terminals or use start_local.sh)
python -m agents.retrieval_agent     # port 8002
python -m agents.orchestrator_agent  # port 8001
python -m agents.main_agent          # port 8000

# 6. Test
python scripts/test_integration.py

# Or directly:
curl -X POST http://localhost:8000/query \
  -H "Content-Type: application/json" \
  -d '{"text": "What is the maternity leave policy?", "conversation_id": "c1", "user_id": "alice"}'
```

## Memory Behaviour

### Short-term memory
- Automatically injected into the synthesis prompt for every query.
- Contains the last `SHORT_TERM_MEMORY_TURNS` (default: 5) turns from the same `conversation_id`.
- Enables follow-up questions like *"What about part-time employees?"* to resolve correctly.
- **Requires Cosmos DB.** Degrades gracefully (no memory) if Cosmos not configured.

### Long-term memory
- After every successful answer, the LLM extracts user-specific facts (e.g. *"User is in HR"*).
- Facts are stored in the `memory` container and injected into future classify prompts.
- Improves domain classification accuracy for returning users.
- **Fire-and-forget** — never blocks the answer path. Failures are logged only.
- View via `GET /memory/{user_id}`.

## MAF Concepts Used

| Concept | Where used |
|---|---|
| `@workflow` | All 3 agents — converts async function to `FunctionalWorkflow` |
| `@step` | All expensive calls (LLM classify, retrieval, synthesis) — adds caching + observability |
| `WorkflowRunResult.get_outputs()` | HTTP endpoints extract typed outputs |
| Functional control flow (`for` loop, `asyncio.gather`) | Orchestrator retry loop, decomposition fan-out |
| `asyncio.create_task` | Fire-and-forget Cosmos writes + memory extraction |

## Graceful Degradation

Cosmos DB is **optional** for local development. If `AZURE_COSMOS_ENDPOINT` or
`AZURE_COSMOS_KEY` are not set, all storage operations are silently skipped:
- `/query` still works — no conversation saved
- `/feedback` returns 500 with a clear message
- `/history` returns empty list
- `/memory` returns empty list
- Memory injection is skipped (no context added)

## Switching to Service Bus in Production

Set `RETRIEVAL_MODE=servicebus` in your environment.
The Orchestrator will use `shared/service_bus.py` instead of direct HTTP.
The Retrieval Agent must run a Service Bus listener (add a background task to its lifespan).
